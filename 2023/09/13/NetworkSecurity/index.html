  • 加密步骤：

    1. HASH(M) => digest
    2. RSA(digest，private密钥) => 数字签名sig
    3. RSA是一种用于签名的公钥加密方法。
    4. 数字签名sig || M => 证书
    5. 压缩(证书) => 压缩证书
    6. AES(会话密钥，压缩证书) => 加密压缩证书（使用对称加密）
    7. RSA(会话密钥，接收者公钥) => 加密会话密钥，用于传输会话密钥非对成的加密
    8. 另一个用户可以使用其私钥获取会话密钥128bit
    9. 发送给接收者：加密压缩证书 || 加密会话密钥

  • 解密步骤：

    1. 使用私钥获取解密的会话密钥 => 会话密钥
    2. 使用会话密钥解密压缩证书 => 压缩证书
    3. 通过解压缩数据获取 M || 数字签名 => M || 数字签名 = DeCompress(压缩证书)
    4. 验证 Hash(M) 和 Hash(M’)
    5. Hash(M’) = 使用发送者的公钥解密数字签名。

  • 压缩：

    • PGP在应用签名之后、加密之前使用ZIP压缩算法。

  • 为什么这个顺序很重要？

    • 签名:最好对未压缩的消息进行签名，以确保签名基于原始的消息。
    • 传输:压缩可以节省电子邮件传输和文件存储空间。
    • 加密up:在压缩之后进行加密可以加强加密，因为压缩减少了消息中的冗余。

IPsec: Internet Protocol Security

IPSec是一种用于保护网络层数据的安全协议。它在互联网层对IP数据报进行安全处理，根据通信IP节点的安全策略，在将数据报转发到网络接口层之前对其进行保护。

接收方IP节点根据建立的安全参数验证数据报，拒绝那些没有按照定义的策略保护的数据报。

IPSec提供的服务包括：Authentication Integrity Confidentiality No-Repulication BUT不会隐藏身份

  1. 身份验证：验证数据源的声明身份，确保发送方的真实性。
  2. 完整性：
    1. 完整性
    2. 抗重放 它可以检测单个IP数据报的篡改和重复到达的数据报。
  3. 保密性：保护数据免受未经授权的披露，并提供有限的流量保密性。它隐藏了源IP地址、目标IP地址、IP数据报的大小和通信频率。

IPsec的整个过程由五个步骤组成：

  1. 启动：有些东西必须触发我们隧道的创建。例如，当我们在路由器上配置IPsec时，我们使用访问列表来告诉路由器要保护哪些数据。当路由器收到与访问列表匹配的内容时，它将启动IKE进程。也可以手动启动隧道。
  2. IKE Phase 1：我们谈判一个security association来建造IKE第1阶段隧道（ISAKMP隧道）。
  3. IKE Phase 2：在IKE第1阶段隧道中，我们构建了IKE第2阶段隧道（IPsec隧道）。
  4. 数据传输：我们通过IKE第2阶段隧道发送用户数据来保护用户数据。
  5. 终止：当没有用户数据需要保护时，IPsec隧道将在一段时间后终止。

IPSec有两种主要的协议：

  1. 身份验证头部（AH）：

    1. INT, data-origin AUTH, anti-replay, access control, no CONF

  2. 封装安全负载（ESP）：

    INT, data-origin AUTH anti-replay, CONF

    ESP can be used to provide only CONF, or AUTH or both.

IPSec有两种模式：Encapsulation Mode

  1. 传输模式：
    1. 在IP数据包中插入IPsec头部，对数据进行保护。Host to host
  2. 隧道模式：
    1. 保留原始数据包，添加新的头部。ESP/AH头部紧跟在新的IP头部之后，原始IP数据包保留不变。gateway to gateway

IPSec架构

  • Authentication header（AH）：用于消息认证的扩展头部。
  • Encapsulating Security Payload（ESP）：提供组合加密和消息完整性的加密。
  • Internet ke exchange（IKE）：用于IPSec的密钥管理方案。
    1. 安全关联（Security Association）的建立：IKE 协议的参与者首先建立安全关联，以定义 IPsec 会话的安全参数，如加密算法、认证算法、密钥长度等。这些参数将用于加密和认证 IP 数据包。
    2. 密钥交换：IKE 使用 Diffie-Hellman 密钥交换算法来协商生成共享密钥。通过 Diffie-Hellman 密钥交换，通信双方可以在不直接传输密钥的情况下，安全地生成共享密钥。
    3. 身份验证：IKE 协议支持多种身份验证方法，如预共享密钥、数字证书、公共密钥基础设施（PKI）等。身份验证确保通信双方可以互相验证对方的身份，并确保没有中间人攻击。
    4. 密钥协商和建立：通过协商过程，IKE 双方使用 Diffie-Hellman 密钥交换算法生成共享密钥，并使用这些密钥来建立 IPsec 会话所需的各种密钥，如对称密钥和非对称密钥

Anonymising Network

C. IPsec不提供在互联网上的匿名性。它主要用于加密和保护数据的安全传输，而不是隐藏或匿名发送方或接收方的身份。

Solution: decentralised, anonymous network

A. 当使用ESP（Encapsulating Security Payload）时，它提供保密性。ESP是IPsec协议套件中的一部分，用于在IP网络上提供数据的加密和认证。

B. IPsec提供了反重播属性，这意味着它可以防止攻击者重复发送已捕获的数据包，从而保护通信的完整性和安全性。

D. 当使用AH（Authentication Header）时，IPsec提供完整性。AH也是IPsec协议套件的一部分，它提供了数据的完整性和身份验证，但不提供加密。No CONF

TOR分布式匿名通信服务

发件人必须与每个路由器协商加密密钥

Tor的组件

  • 客户端
  • 服务器
  • Tor（洋葱）路由器：特殊代理中继应用数据。
  • 目录服务器：提供Tor路由器信息。Directory server

过程

  • 最后一个路由器将看到明文数据，但不知道消息来自哪里。
  • 每个路由器只知道前驱和后继。

客户端发送消息的过程：

  1. 客户端从目录服务器获取Tor节点列表。
  2. 客户端随机选择一条到目标服务器的路径。
  3. 客户端与每个路由器协商AES密钥（每个路由器都有自己的加密密钥）。
  4. 客户端加密消息：
    • C3 = Encrypt(K3, data||IP_Server)
    • C2 = Encrypt(K2, C3 || IP_OR3)
    • C1 = Encrypt(K1, C2 || IP_OR2)
  5. 客户端发送一个IP数据包：IP_Client || IP_OR1 || C1

每个路由器的数据包处理过程：

  • 数据包到达OR1：
    • C2 || IP_OR2 = DEC(K1, C1)
    • 缓存IP_Client，IP_OR2
    • 发送一个IP数据包：IP_OR1 || IP_OR2 || C2
  • 数据包到达OR2：
    • C3 || IP_OR3 = DEC(K2, C2)
    • 缓存IP_OR1 || IP_OR3
    • 发送一个IP数据包：IP_OR2 || IP_OR3 || C3
  • 数据包到达OR3：
    • IP_Server || data = DEC(K3,C3)
    • 缓存IP_OR2，IP_Server
    • 发送一个IP数据包：IP_OR3 || IP_SERVER || data

Requirements of a firewall

  • All the traffic between trust zones should pass through the firewall.
  • Only authorized traffic, as defined by the security policy, should be allowed to pass through.
  • The firewall itself must be immune to penetration, which implies using a hardened system with secured Operating Systems.
  • Determines the direction in which requests may be initiated and are allowed to flow through the firewall. It tells whether the traffic is “inbound” (From the network to the firewall) or vice-versa “outbound”

Firewall Policy

• User control:

Controls access to the data based on the role of the user who is attempting to access it. Applied to users inside the firewall perimeter.
• Service control:

Controls access by the type of service offered by the host. Applied on the basis of network address, the protocol of connection and port numbers.
• Direction control:

Determines the direction in which requests may be initiated and are allowed to flow through the firewall. It tells whether the traffic is “inbound” (From the network to firewall) or vice-versa “outbound”

Overview of TLS

Transport Layer Security (TLS) provides a secure channel between two communicating applications.

  • Integrity: Channel can detect any changes made to the data during transmission
  • Authentication: At least one end of the channel needs to be authenticated, so the other end knows who it is talking to.
  • Confidentiality: Nobody other than the two ends of the channel can see the actual content of the data transmitted.

传输层安全（TLS）在两个通信应用程序之间提供了一个安全通道。

  • 完整性：通道可以检测传输过程中对数据所做的任何更改
  • 身份验证：通道的至少一端需要进行身份验证，因此另一端知道它正在与谁交谈。
  • 保密性：除了频道的两端，没有人能看到所传输数据的实际内容。

填充预言攻击的目标是破解加密数据的机密性，特别是针对使用分组密码模式（如CBC模式）进行加密的情况。

在这种模式下，明文数据被分割为固定大小的块，并使用密钥加密后传输。每个加密块都会使用前一个加密块的密文作为初始向量（IV）进行加密。

填充预言攻击的步骤如下：

  1. 攻击者截获加密的密文数据，并选择一个加密块进行攻击。
  2. 攻击者修改该加密块的最后一个字节，并将修改后的密文发送回解密的实体。
  3. 当解密实体尝试解密修改后的密文时，如果填充验证步骤失败（例如，填充字节的格式不正确），则解密实体将返回一个错误，标识填充验证失败。
  4. 攻击者利用填充验证失败的结果来推断出修改后的密文块的最后一个字节的真实值。这是因为填充验证步骤会提供关于填充字节格式正确与否的信息，攻击者可以通过不断修改填充字节，直到填充验证成功，从而推断出填充字节的正确值。
  5. 通过逐个字节地重复上述步骤，攻击者可以一次推断出加密块中的每个字节的值。

Countermeasures

  • Remove server responses
  • Use AES GCM model - authenticated encryption
  • Ensure that the ciphertext is not modified during the transmission
  • 删除服务器响应
  • 使用AES GCM模型-经过身份验证的加密

MAC地址（Media Access Control Address）是网络接口的唯一标识符，用于在局域网中识别设备。然而，基于MAC地址的身份验证可以被绕过，因为MAC地址可以被欺骗或伪造。

攻击者可以使用特定的软件或工具来修改其设备的MAC地址，以伪装成受信任的设备。这种欺骗可以使攻击者绕过基于MAC地址的身份验证措施，获取未经授权的网络访问权限。

因此，尽管MAC地址是设备的唯一标识符，但它并不是一个可靠的身份验证机制，因为它可以被篡改或伪造。在无线网络中，为了更强大的安全性，通常需要使用更高级的身份验证方法，例如WPA/WPA2密码、企业级802.1X认证等。

IEEE 802.11b中WEP的安全性（Security of WEP）：

IEEE 802.11b中WEP的不安全性（Insecurity of WEP）：

  1. 密钥弱度：WEP使用的40位或104位密钥空间相对较小，容易受到穷举攻击。攻击者可以通过尝试所有可能的密钥组合来破解WEP加密。
  2. 初始化向量（IV）的重用：WEP使用24位的初始化向量来增加加密强度。然而，WEP在实际使用中重复使用相同的IV，导致加密流量的重复性，从而使攻击者能够获取加密密钥。
  3. RC4算法弱点：WEP使用的RC4算法存在一些弱点，使得攻击者能够通过分析加密的数据流来恢复密钥。这种攻击称为WEP密钥恢复攻击。
  4. 数据完整性问题：WEP未提供数据完整性检查机制，这意味着攻击者可以对数据进行篡改而不被检测到。
  5. 密钥管理和认证问题：WEP采用了弱密钥管理和认证机制，容易受到字典攻击和伪造认证攻击。

  • 完整性（Integrity）：通过校验和和消息认证码（MAC）来确保数据在传输过程中没有被篡改。

TKIP（Temporal Key Integrity Protocol）和CCMP（Counter mode with Cipher Block Chaining Message Authentication Code Protocol）的区别：

  • TKIP是为了改进WEP（Wired Equivalent Privacy）的安全性而引入的。它提供了加密、完整性和消息认证，但相对较弱。
  • CCMP是在WPA2中使用的加密协议，提供了认证、完整性和机密性，比TKIP更安全。

应用层：

  1. SYN洪水攻击：

    • SYN洪水攻击利用TCP三次握手过程中的漏洞，攻击者通过持续发送大量伪造的SYN数据包给服务器，耗尽服务器队列中的**资源TCB (Transmission Control Block)**。

    对策：SYN Cookie

    • 收到 SYN 数据包后，服务器将使用密钥根据 SYN 数据包计算密钥哈希。
    • 该哈希值 (H) 作为初始序列号从服务器发送到客户端（IP 地址）。H称为SYN cookie。
    • 服务器不会将半打开连接存储在其队列中。
    • 由于IP地址是假的，攻击者不会收到H。
    • 正常的客户端会发送H+1给服务器，服务器会检查它。

    The server calculates a keyed hash upon receiving a SYN packet. This hash is then sent to the client. If acknowledgment of this hash is not responded to, the server does not store 2 half-open connections in its queue.

  2. TCP复位攻击：

    • TCP复位攻击通过发送RST数据包来立即中断两个主机之间的连接。
    • TLS无法防止TCP复位攻击，因为TLS位于TCP之上，无法保护TCP头部，所以仍然可以设置RST标志位。
    • 解决方法：使用入侵检测系统（IDS）来检测和阻止TCP复位攻击。

  3. TCP会话劫持攻击：

    • 黑客可以通过捕获数据包并next sequence number和acknowledgement number来执行TCP会话劫持攻击。

    • 解决方法：

      • 难以伪造数据包：随机化初始序号和源端口号。
      • 对TCP负载进行加密。

      Making it difficult for attackers to spoof packets
      • Randomise source port number
      • Randomise initial sequence number
      • Encrypting payload

  2. 本地DNS缓存中毒攻击：
    • 针对用户机器的攻击：攻击者向发送DNS查询的用户机器发送带有恶意IP地址的伪造DNS响应。
